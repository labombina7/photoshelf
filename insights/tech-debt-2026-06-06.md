## Hallazgos

### 🔒 Seguridad

#### Endpoint `/api/photos` sin límite superior en el parámetro `limit`
- **Fichero:** `src/app/api/photos/route.ts:21`
- **Severidad:** 🟡 Media
- **Descripción:** `limit` se lee del querystring sin cota máxima. Un cliente (o el propio cliente legítimo con `limit=5000` para el slideshow) puede forzar una query que retorne decenas de miles de filas y sus tags asociados en una sola respuesta, agotando memoria y CPU del servidor SQLite.
- **Fix propuesto:** Aplicar `Math.min(limit, MAX_LIMIT)` con `MAX_LIMIT = 1000` (o similar). Para el slideshow recuperar solo `id` en lugar de el objeto completo.

#### `deleteCatalog` ejecuta dos DELETEs sin transacción
- **Fichero:** `src/lib/queries/catalogs.ts` (función `deleteCatalog`)
- **Severidad:** 🔴 Alta
- **Descripción:** Si el proceso muere entre el `DELETE FROM photos` y el `DELETE FROM catalogs`, el catálogo permanece en BD sin fotos. Peor aún: si se elimina el registro del catálogo pero alguna foto ha sido ya borrada, no hay rollback. Tampoco protege contra ejecuciones concurrentes del mismo endpoint.
- **Fix propuesto:** Envolver ambas sentencias en `db.transaction(...)`.

#### Race condition en `scanState` / `watcherState` sin bloqueo de concurrencia
- **Fichero:** `src/lib/scanState.ts`, `src/lib/folderWatcher.ts`
- **Severidad:** 🔴 Alta
- **Descripción:** El estado de escaneo se comparte mediante `globalThis` sin ningún mecanismo de exclusión mutua. Si dos peticiones POST `/api/scan` llegan simultáneamente y superan la comprobación `getScanState().running` antes de que la primera actualice el estado, se lanzan dos escaneos paralelos que insertan registros duplicados (el `ON CONFLICT` los resolverá, pero se consume el doble de recursos). La comprobación + actualización no es atómica.
- **Fix propuesto:** Usar una Promise-guard o un flag síncrono con asignación antes del `after()`. En rutas que modifican `scanState` hacer check + update en la misma función sincrónica.

### ⚠️ Gestión de errores

#### `catch {}` silencioso en decodificación de cursor (timeline v1)
- **Fichero:** `src/app/api/v1/timeline/[periodKey]/photos/route.ts:42`
- **Severidad:** 🟢 Baja
- **Descripción:** Si el cursor base64 llega corrupto o truncado, la excepción se silencia y `decodedCursor` queda `null`, devolviendo los primeros resultados sin cursor. El comportamiento es razonablemente seguro pero el error no queda registrado, dificultando el diagnóstico de clientes buggeados.
- **Fix propuesto:** Añadir `console.warn('[timeline-photos] Invalid cursor, ignoring:', cursor)` dentro del catch.

#### `catch {}` silencioso en timeline v1 (cursor de períodos)
- **Fichero:** `src/app/api/v1/timeline/route.ts:40`
- **Severidad:** 🟢 Baja
- **Descripción:** Idéntico al anterior, mismo patrón. Fácil de corregir junto con el de arriba.
- **Fix propuesto:** Añadir log de warning en el catch.

#### `catch` sin log en `v1/photos/[id]/original` (path traversal)
- **Fichero:** `src/app/api/v1/photos/[id]/original/route.ts:44`
- **Severidad:** 🟢 Baja
- **Descripción:** Cuando `resolvePhotoPath` lanza "Path traversal detected" el catch llama a `apiError` pero no loguea. Cualquier intento real de path traversal pasa desapercibido en los logs.
- **Fix propuesto:** Añadir `console.error('[security] Path traversal attempt, photo id:', id)` antes de `apiError`.

### 🏷️ Tipos TypeScript

#### `MIME_TYPES` duplicado en `v1/photos/[id]/original/route.ts`
- **Fichero:** `src/app/api/v1/photos/[id]/original/route.ts:10-22`
- **Severidad:** 🟡 Media
- **Descripción:** El route de la API v1 define su propio objeto `MIME_TYPES` local en lugar de importar desde `src/lib/config.ts`. El original tiene 9 entradas, la copia tiene 10 (incluye `.avif`). La divergencia ya ha causado una inconsistencia: `config.ts` no contempla `.avif` mientras que este route sí.
- **Fix propuesto:** Eliminar la copia local. Añadir `.avif` a `MIME_TYPES` en `config.ts` e importar desde ahí.

#### `OLLAMA_URL` duplicado fuera de `config.ts`
- **Fichero:** `src/app/api/ollama/status/route.ts:4` y `src/lib/ollama/client.ts:3`
- **Severidad:** 🟡 Media
- **Descripción:** `OLLAMA_URL` se define con `process.env.OLLAMA_URL ?? 'http://localhost:11434'` en dos sitios distintos. Si se cambia el fallback en uno, el otro queda desincronizado.
- **Fix propuesto:** Mover la constante a `config.ts` y exportarla desde allí.

#### Dead code: `src/lib/queries/catalog.ts` (stub pre-EPIC-001)
- **Fichero:** `src/lib/queries/catalog.ts`
- **Severidad:** 🟢 Baja
- **Descripción:** El fichero `catalog.ts` contiene `getDefaultCatalog()` que retorna siempre `{ id: 1 }`. EPIC-001 ya está implementado y ningún fichero importa esta función. Es código muerto que puede generar confusión.
- **Fix propuesto:** Eliminar el fichero (verificar con `grep -r "getDefaultCatalog"` que no se usa).

### 🚀 Rendimiento

#### Query de `listPhotos` ejecuta COUNT(*) siempre, incluso si no es necesario
- **Fichero:** `src/lib/queries/photos.ts:61-67`
- **Severidad:** 🟢 Baja
- **Descripción:** En cada petición de listado se hacen 3 queries: SELECT DISTINCT, COUNT y SELECT DISTINCT years. Para catálogos con cientos de miles de fotos el COUNT con JOINs puede ser costoso. No hay paginación cursor-based como en la API v1.
- **Fix propuesto:** Considerar añadir un parámetro `skipTotal` para cuando el cliente sólo necesita las fotos (ej. scroll infinito). La API v1 ya implementa cursor-based pagination correctamente.

#### Thumbnails se generan síncronamente en el hilo principal del worker
- **Fichero:** `src/lib/thumbnail.ts` (referenciado desde `src/lib/worker.ts`)
- **Severidad:** 🟢 Baja
- **Descripción:** Los thumbnails se generan con `sharp` de forma secuencial dentro del worker. No hay pool de workers de imagen, lo que puede bloquear el event loop durante operaciones de clasificación masiva con muchas fotos HEIC (requieren conversión previa).
- **Fix propuesto:** No urgente dada la escala prevista, pero documentar el cuello de botella para cuando escale.

### 🏗️ Arquitectura

#### `LibraryClient.tsx` y `DetailPanel.tsx` superan las 300 líneas
- **Fichero:** `src/app/library/LibraryClient.tsx` (398 líneas), `src/components/DetailPanel.tsx` (396 líneas), `src/app/search/SearchClient.tsx` (367 líneas)
- **Severidad:** 🟡 Media
- **Descripción:** Tres componentes superan las 300 líneas mezclando lógica de negocio (fetch, estado, validación) con presentación. Dificultan el testing unitario y el mantenimiento.
- **Fix propuesto:** Extraer hooks custom (`usePhotoFilter`, `useTagEditor`, `useAiReview`) y subcomponentes de presentación.

#### `scan/route.ts` usa `getDb()` directamente (viola convención de repositorio)
- **Fichero:** `src/app/api/photos/[id]/original/route.ts:14`, `src/app/api/photos/[id]/thumbnail/route.ts:11`
- **Severidad:** 🟡 Media
- **Descripción:** Varios route handlers importan `getDb()` directamente para queries puntuales, saltándose la capa de repositorio en `src/lib/queries/`. Esto contradice la convención documentada en CLAUDE.md.
- **Fix propuesto:** Mover las queries inline (`SELECT p.path, p.filename ... WHERE p.id = ?`) a funciones en `src/lib/queries/photos.ts`.

### 🔧 Mantenibilidad

#### 3 `eslint-disable react-hooks/exhaustive-deps` en `LibraryClient.tsx`
- **Fichero:** `src/app/library/LibraryClient.tsx:139, 209, 249`
- **Severidad:** 🟡 Media
- **Descripción:** Las supresiones ocultan dependencias de `useEffect` y `useMemo` que podrían causar stale closures. En particular, `openSlideshow` se usa dentro del `useEffect` del teclado pero no está en las dependencias (suprimir evita el re-registro del listener, pero la función captura `activeFilters` del momento del mount).
- **Fix propuesto:** Usar `useCallback` para estabilizar `openSlideshow` y declararlo como dependencia explícita, o usar un ref para acceder a los filtros actuales dentro del listener.

#### `checkPassword` tiene un `try/catch` vacío (timing attack defense incompleta)
- **Fichero:** `src/lib/session.ts:41-47`
- **Severidad:** 🟡 Media
- **Descripción:** Cuando la contraseña esperada está vacía o las longitudes difieren, se hace una comparación de relleno para evitar timing attacks, pero el `try/catch` vacío ignora cualquier excepción de `timingSafeEqual`. Si `Buffer.from` lanza (ej. input con caracteres problemáticos), el error se silencia y la función retorna `false` silenciosamente.
- **Fix propuesto:** Al menos añadir `console.error` en el catch, o simplificar: si `!expected` devolver siempre false sin comparación (el timing attack de longitud es aceptable cuando el secret está vacío porque la config ya está mal).

### 🧪 Testing

#### `deleteCatalog` no tiene tests
- **Fichero:** `src/lib/queries/catalogs.ts`
- **Severidad:** 🔴 Alta
- **Descripción:** La operación más destructiva del sistema (borrar un catálogo con todas sus fotos) no tiene ningún test. Tampoco la ausencia de transacción está cubierta por tests de concurrencia.
- **Fix propuesto:** Añadir tests en `src/app/api/catalogs/__tests__/catalogs.test.ts` que cubran: borrado exitoso, intento de borrar catálogo 1, intento de borrar catálogo activo, catálogo inexistente.

#### `backup.ts` y `folderWatcher.ts` sin cobertura
- **Fichero:** `src/lib/backup.ts`, `src/lib/folderWatcher.ts`
- **Severidad:** 🟡 Media
- **Descripción:** Las funciones de backup y watcher son críticas para la integridad de datos y no tienen tests. El watcher en particular tiene lógica de debounce y snapshot de directorios que sería valiosa de testear.
- **Fix propuesto:** Tests de backup con mocks de `fs` y `getDb()`. Tests del watcher verificando que `scheduleAutoScan` se llama tras detectar nuevos directorios.

#### `integrityScanner.ts` sin tests
- **Fichero:** `src/lib/integrityScanner.ts`
- **Severidad:** 🟡 Media
- **Descripción:** El scanner de integridad detecta fotos huérfanas y sin indexar — no tiene tests que verifiquen su lógica de clasificación.
- **Fix propuesto:** Tests unitarios con un sistema de ficheros mockeado.

### 💾 Riesgos de pérdida de datos

#### `deleteCatalog` sin transacción (duplicado de Seguridad)
- **Fichero:** `src/lib/queries/catalogs.ts`
- **Severidad:** 🔴 Alta
- **Descripción:** Los dos DELETEs (fotos y catálogo) no están envueltos en `db.transaction()`. Un crash entre ambas sentencias deja los datos en estado inconsistente. Las fotos se borran permanentemente (no hay soft-delete).
- **Fix propuesto:** `db.transaction(() => { db.prepare(...).run(id); db.prepare(...).run(id); })()`.

#### Borrado de fotos es destructivo en BD pero no toca el disco
- **Fichero:** `src/lib/queries/catalogs.ts`, `src/app/api/photos/[id]/route.ts` (si existe DELETE)
- **Severidad:** 🟡 Media
- **Descripción:** Al borrar un catálogo se borran los registros de fotos de la BD pero los ficheros físicos permanecen en el NAS (que está montado en solo lectura, por lo que esto es correcto). Sin embargo, los thumbnails generados en `CACHE_PATH` para esas fotos quedan huérfanos y no hay limpieza automática.
- **Fix propuesto:** Documentar el comportamiento. Opcionalmente, añadir una tarea de limpieza de caché huérfana al integrity scanner.

#### `scanState` no persiste entre reinicios del servidor
- **Fichero:** `src/lib/scanState.ts`
- **Severidad:** 🟢 Baja
- **Descripción:** Si el servidor se reinicia durante un escaneo, `running` queda como `false` en `globalThis` (ya que se reinicializa), pero el escaneo no terminó. Las fotos parcialmente indexadas permanecen en BD con `scanned_at` actualizado. En el siguiente escaneo el `ON CONFLICT DO UPDATE` las corregirá, por lo que el riesgo real es bajo.
- **Fix propuesto:** Documentar el comportamiento. No requiere acción inmediata.

### 🐳 Persistencia en Docker

#### Thumbnails y BD correctamente en el volumen `/data`
- **Severidad:** 🟢 Baja (positivo)
- **Descripción:** `DB_PATH=/data/photoshelf.db`, `CACHE_PATH=/data/.cache` y `BACKUP_PATH=/data/backups` apuntan todos al volumen montado. El NAS completo está en solo lectura. La configuración es correcta.

#### No hay `healthcheck` en el `docker-compose.yml`
- **Fichero:** `docker-compose.yml`
- **Severidad:** 🟢 Baja
- **Descripción:** Sin `healthcheck`, el orquestador (Docker o Synology Container Manager) no puede detectar si la app ha arrancado correctamente o si está en estado zombie. Solo sabe si el proceso existe.
- **Fix propuesto:**
  ```yaml
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:3000/api/health"]
    interval: 30s
    timeout: 5s
    retries: 3
    start_period: 15s
  ```

#### `node-addon-api` en `dependencies` engrosa imagen de producción
- **Fichero:** `package.json`
- **Severidad:** 🟢 Baja
- **Descripción:** `node-addon-api` es solo necesario en build time (para compilar `better-sqlite3`). Al estar en `dependencies` se copia a la imagen final, añadiendo ~1MB innecesario.
- **Fix propuesto:** Mover a `devDependencies`.

#### WAL mode habilitado — correcto, pero sin backup automático activado por defecto
- **Fichero:** `src/lib/db.ts:12`, `src/lib/queries/backup.ts`
- **Severidad:** 🟡 Media
- **Descripción:** `journal_mode = WAL` está correctamente configurado. El sistema de backup manual existe y funciona. Sin embargo, `auto_enabled = 0` por defecto — un usuario que instale la app sin leer la documentación nunca tendrá backups automáticos.
- **Fix propuesto:** Cambiar el default de `auto_enabled` a `1` con `auto_interval_days = 7`, o mostrar un aviso en la UI de ajustes si el backup automático está desactivado.
