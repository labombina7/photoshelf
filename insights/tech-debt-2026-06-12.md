**Contexto positivo:** la base está sana — WAL activado, path traversal protegido (`resolvePhotoPath`), comparación de contraseña timing-safe, validación de env vars al arrancar (`instrumentation.ts`), migraciones idempotentes bien documentadas, `deleteCatalog` transaccional, backup atómico con `VACUUM INTO` + export JSON de tags. ## Hallazgos

### 🔒 Seguridad

#### Parámetro `size` del thumbnail sin validar ni acotar
- **Fichero:** `src/app/api/photos/[id]/thumbnail/route.ts:15`
- **Severidad:** 🟡 Media
- **Descripción:** `parseInt(sp.get('size') ?? '400')` se pasa directo a sharp sin clamp ni check de NaN. Un `size=20000` fuerza redimensionados enormes (CPU + memoria) y cada valor distinto genera una entrada nueva en la caché de disco — vector de agotamiento de recursos para cualquier sesión autenticada (o vía `<img>` CSRF).
- **Fix propuesto:** Whitelist de tamaños (`[100,150,200,300,400,420,600,1920]`) o clamp `Math.min(Math.max(size||400, 50), 1920)` antes de llamar a `getThumbnail`.

#### Datos personales hardcodeados en docker-compose.yml comiteado
- **Fichero:** `docker-compose.yml:10,24`
- **Severidad:** 🟢 Baja
- **Descripción:** `PHOTOS_PATH: /volume1/homes/javi/Photos`, IP de Ollama `192.168.1.135` y `DATA_PATH` por defecto con ruta personal están en el repo (público en ghcr.io/labombina7). No son secretos, pero exponen topología de red doméstica y rompen la portabilidad del compose.
- **Fix propuesto:** Mover a variables `${PHOTOS_PATH}` con defaults genéricos y documentar en `.env.example`.

### ⚠️ Gestión de errores

#### `toggleFavorite` del DetailPanel sin manejo de error ni revert
- **Fichero:** `src/components/DetailPanel.tsx:131-140`
- **Severidad:** 🟡 Media
- **Descripción:** El PATCH es fire-and-forget: sin try/catch, sin revertir el estado optimista. Si la API falla, la UI muestra un favorito que no se guardó. La versión de `PhotoGrid.tsx:228-267` sí implementa revert — patrón inconsistente entre componentes.
- **Fix propuesto:** Copiar el patrón de PhotoGrid (revert en catch) y reutilizar el `errorToast` que el panel ya tiene para tags.

#### Polling de generación de proyectos sin timeout ni límite de reintentos
- **Fichero:** `src/app/projects/ProjectsClient.tsx:109-128`
- **Severidad:** 🟡 Media
- **Descripción:** `setInterval` cada 2 s para el job de Ollama sin condición de salida temporal. Si el job queda colgado (servidor reiniciado con worker sin arrancar — ver hallazgo de jobs), el modal queda en "Generando…" para siempre y el interval vive hasta cerrar la pestaña.
- **Fix propuesto:** Contador de intentos (~90 = 3 min) con mensaje de error y limpieza del interval en el unmount del componente (`useEffect` cleanup — ahora mismo el interval ni siquiera se limpia si el usuario navega).

#### Validación de paginación después de usarla
- **Fichero:** `src/app/api/v1/photos/route.ts:20-28`
- **Severidad:** 🟢 Baja
- **Descripción:** `if (isNaN(limit)) return apiError(...)` se evalúa **después** de llamar a `listPhotos()` con ese limit — la query ya se ejecutó con NaN (SQLite lo trata como NULL → comportamiento indefinido del LIMIT).
- **Fix propuesto:** Mover el check antes de `listPhotos`.

#### `handleClassify` silencia errores de red sin feedback
- **Fichero:** `src/components/PhotoGrid.tsx:312-314`
- **Severidad:** 🟢 Baja
- **Descripción:** El catch solo hace `setClassifying(false)` — si el POST a `/api/ai/classify/batch` falla, el usuario no ve ningún mensaje.
- **Fix propuesto:** Setear `classifyResult` con un mensaje de error o reutilizar el toast.

### 🏷️ Tipos TypeScript

#### `require('archiver') as any` en runtime ESM
- **Fichero:** `src/app/share/[token]/route.ts:76`
- **Severidad:** 🟢 Baja
- **Descripción:** `new (require('archiver') as any).Archiver(...)` evita el sistema de tipos y usa `require` con eslint-disable en un módulo ESM. Archiver tiene tipos (`@types/archiver` ya instalado).
- **Fix propuesto:** `import archiver from 'archiver'` + `archiver('zip', { zlib: { level: 0 } })` (y añadir `archiver` a `serverExternalPackages` si el bundler da problemas).

#### Interfaces duplicadas entre componentes
- **Fichero:** `src/app/library/LibraryClient.tsx:19-38` vs `src/components/PhotoGrid.tsx:31-49` (`EventGroup`, `ActiveFilters`)
- **Severidad:** 🟢 Baja
- **Descripción:** `EventGroup` y `ActiveFilters` están definidos dos veces con campos casi idénticos (uno tiene `thumbnail_id`, otro `tag`). También se repite el patrón `import('@/lib/queries/catalogs').CatalogRow` inline en 6+ ficheros.
- **Fix propuesto:** Mover ambos a `src/lib/types.ts` y exportar `CatalogRow` desde ahí.

### 🚀 Rendimiento

#### Cliente pide `limit=2000` pero el servidor capa a 1000 — truncado silencioso
- **Fichero:** `src/components/PhotoGrid.tsx:151` vs `src/app/api/photos/route.ts:22` (`PHOTOS_MAX_LIMIT = 1000`)
- **Severidad:** 🟡 Media
- **Descripción:** PhotoGrid pide hasta 2000 fotos por evento en una sola petición; el handler hace `Math.min(limit, 1000)`. Un evento con >1000 fotos muestra solo 1000 sin ningún aviso, y además la respuesta de ~1000 fotos con tags es un payload pesado de golpe.
- **Fix propuesto:** Paginar con offset/cursor desde PhotoGrid (el render ya es incremental con `visible`) o al menos detectar `total > photos.length` y avisar/encadenar peticiones.

#### Polling continuo de `/api/jobs` cada 3 segundos en el sidebar
- **Fichero:** `src/components/Sidebar.tsx:410-423` (`ToolsSection`)
- **Severidad:** 🟢 Baja
- **Descripción:** Mientras el módulo Herramientas está visible, se golpea `/api/jobs` cada 3 s aunque no haya ningún job activo. Además `/api/health` (GET) escribe un snapshot en BD en cada visita.
- **Fix propuesto:** Backoff cuando la respuesta no tiene jobs activos (3 s → 15 s) y pausar con `document.visibilityState`.

#### `getThumbnail` sin deduplicación de generaciones concurrentes
- **Fichero:** `src/lib/thumbnail.ts:39-92`
- **Severidad:** 🟢 Baja
- **Descripción:** Si N requests piden el mismo thumbnail no cacheado a la vez (primer render de un grid), sharp lo genera N veces y escriben el mismo fichero.
- **Fix propuesto:** Mapa de promesas in-flight por cacheKey (`pending.get(key) ?? generate()`).

#### `allPhotos.indexOf(photo)` dentro del render del timeline — O(n²)
- **Fichero:** `src/app/timeline/TimelineClient.tsx:367`
- **Severidad:** 🟢 Baja
- **Descripción:** Para decidir `fetchPriority`, cada foto renderizada busca su índice en el array completo (miles de elementos tras varios scrolls).
- **Fix propuesto:** Pasar el índice global precalculado al agrupar, o limitar el cálculo a la primera página.

### 🏗️ Arquitectura

#### Doble superficie de API: `/api/*` y `/api/v1/*`, la v1 sin consumidores
- **Fichero:** `src/app/api/v1/**` (10 routes) vs `src/app/api/**` (legacy)
- **Severidad:** 🟡 Media
- **Descripción:** EPIC-002 creó la capa v1 con `withAuth`/`apiSuccess` (patrón superior), pero **ningún componente del frontend la usa** — todos los `fetch()` van a las rutas legacy. Se mantienen dos implementaciones de photos/timeline/catalogs/tags en paralelo, con el riesgo de que diverjan (ya divergen: la v1 pagina con cursor, la legacy con offset).
- **Fix propuesto:** Decidir: migrar el frontend a v1 y borrar las legacy, o eliminar v1 si no hay un cliente externo previsto.

#### El watcher ignora el sistema multi-catálogo
- **Fichero:** `src/lib/folderWatcher.ts:68,91-98,113`
- **Severidad:** 🟡 Media
- **Descripción:** `runAutoScan` escanea siempre `PHOTOS_PATH` (env) hacia `catalogId=1`, aunque el catálogo 1 pueda haber cambiado de ruta vía UI (EPIC-001 lo permite). Peor: `runAutoClassify` selecciona fotos sin tags de **todos** los catálogos pero resuelve sus paths contra `PHOTOS_PATH` — las fotos de otros catálogos fallan la clasificación silenciosamente (catch vacío en línea 115) en cada ciclo.
- **Fix propuesto:** Resolver la ruta del catálogo desde BD (`getCatalogById(1).path`, como ya hace `worker.ts:126`) y filtrar `runAutoClassify` por `catalog_id = 1` (o iterar catálogos).

#### Polling de jobs duplicado en tres componentes
- **Fichero:** `src/components/PhotoGrid.tsx:269-289`, `src/app/projects/ProjectsClient.tsx:109-128`, `src/components/ClassifyProvider.tsx`
- **Severidad:** 🟢 Baja
- **Descripción:** El patrón "setInterval → fetch `/api/jobs/{id}` → actualizar progreso → detectar estado terminal" está copiado tres veces con pequeñas variaciones (y bugs distintos: uno limpia el interval en unmount, otro no).
- **Fix propuesto:** Hook compartido `useJobPolling(jobId, { onComplete, onFail })` en `src/hooks/`.

### 🧪 Testing

#### 21 tests fallando en main (14 suites rojas)
- **Fichero:** `src/__tests__/ollama.test.ts`, `src/lib/__tests__/classifier.test.ts`, `scanner.test.ts`, `thumbnail.test.ts`
- **Severidad:** 🔴 Alta
- **Descripción:** `vitest run` da **223/244 passed, 21 failed**. Dos causas: (1) el mock de sharp no implementa `removeAlpha()` que ahora usa `detectIsBlackAndWhite` (`src/lib/ollama/image.ts:30`) — rompe ollama, scanner y thumbnail tests; (2) los tests de `classifyQuery` esperan el comportamiento antiguo del clasificador de intención (rangos de año, detección por nº de palabras). La suite lleva rota desde algún cambio reciente y nadie lo nota porque CI no ejecuta tests.
- **Fix propuesto:** Actualizar el mock de sharp (añadir `removeAlpha: () => this`) y reescribir los asserts de classifier contra el comportamiento actual. Estimar 2–3 h.

#### CI no ejecuta tests ni lint — solo construye la imagen
- **Fichero:** `.github/workflows/build.yml`
- **Severidad:** 🟡 Media
- **Descripción:** El único workflow hace build+push de Docker en cada push a main. No hay gate de `vitest`, `tsc` ni `next lint` — por eso main tiene 21 tests rojos sin que nada avise. Una imagen rota podría publicarse a `:latest` (que el NAS consume).
- **Fix propuesto:** Job previo `test` (npm ci + vitest run + lint) como `needs:` del build, y considerar tags versionados en vez de solo `:latest`.

#### Sin tests de componentes React
- **Fichero:** `src/components/**` (0 tests), `@testing-library/react` instalado sin uso
- **Severidad:** 🟢 Baja
- **Descripción:** Toda la cobertura es de `lib/` y un par de API routes. La lógica de UI con más bugs históricos (navegación por teclado de PhotoGrid, gestos del visor móvil, FilterBar) no tiene ninguna red de seguridad.

### 💾 Riesgos de pérdida de datos

#### El enlace de compartir se quema antes de completar la descarga
- **Fichero:** `src/app/share/[token]/route.ts:66-67`
- **Severidad:** 🟡 Media
- **Descripción:** `markShareTokenUsed(token)` se ejecuta **antes** de hacer streaming del ZIP. Si la descarga se corta a la mitad (red móvil, ZIP de 300 fotos sin comprimir puede ser >1 GB), el enlace de uso único queda consumido y el destinatario no puede reintentar — hay que regenerar y reenviar el enlace.
- **Fix propuesto:** Marcar como usado al **terminar** el stream (evento `close`/`finish` del archiver con bytes completos), o permitir N reintentos dentro de una ventana corta (p.ej. 3 usos / 1 h tras el primer uso).

#### Jobs `in_progress` quedan congelados tras un reinicio hasta que alguien visita la app
- **Fichero:** `src/lib/worker.ts:46-56` + `src/instrumentation.ts:17-26`
- **Severidad:** 🟡 Media
- **Descripción:** `getNextJob()` sí reanuda jobs `in_progress` (bien diseñado), pero el worker loop solo arranca cuando alguna ruta llama a `ensureWorkerRunning()` (classify, jobs, generate). Tras reiniciar el contenedor, un classify a medias o el auto-backup vencido no se reanudan hasta que un usuario navegue a esas rutas — pueden pasar días en un servidor doméstico.
- **Fix propuesto:** Llamar a `ensureWorkerRunning()` desde `instrumentation.ts` junto al watcher.

#### Backups en el mismo volumen que la base de datos
- **Fichero:** `docker-compose.yml:14` (`BACKUP_PATH=/data/backups`), `src/lib/backup.ts`
- **Severidad:** 🟢 Baja
- **Descripción:** El backup rotado (10 copias) vive en `/data`, el mismo volumen que `photoshelf.db` — protege contra corrupción lógica pero no contra pérdida del volumen/disco. `.env.example` ya recomienda cubrir `/data` con Hyper Backup, lo cual mitiga si se cumple.
- **Fix propuesto:** Documentar en README la dependencia de Hyper Backup, o soportar un segundo destino (`BACKUP_PATH` apuntando a otro volumen montado).

#### Reset de versión de análisis de estilo borra perfiles sin export
- **Fichero:** `src/lib/db.ts:431-438`
- **Severidad:** 🟢 Baja
- **Descripción:** Al subir `STYLE_ANALYSIS_VERSION`, el arranque ejecuta `DELETE FROM style_profiles` de todos los perfiles (incluidas narrativas generadas por Ollama, caras de regenerar). Es recomputable pero costoso (horas de Ollama).
- **Fix propuesto:** Renombrar a `style_profiles_v{N}_backup` o exportar a JSON antes del DELETE.

### 🐳 Persistencia en Docker

#### Healthcheck roto: endpoint autenticado + curl inexistente en la imagen
- **Fichero:** `docker-compose.yml:16-21`, `Dockerfile:16-22`, `src/middleware.ts:20`
- **Severidad:** 🔴 Alta
- **Descripción:** Doble fallo: (1) `node:20-slim` no incluye `curl`, así que el comando del healthcheck falla con "command not found"; (2) aunque existiera, `/api/health` pasa por el middleware que devuelve 401 sin cookie de sesión. Resultado: el contenedor está **permanentemente unhealthy** — cualquier orquestador o monitor que actúe sobre el estado (watchtower, Portainer, dependencias `depends_on: condition: service_healthy`) lo tratará como caído.
- **Fix propuesto:** (a) Excluir un endpoint liviano del middleware (p.ej. `/api/ping` sin BD) añadiéndolo al negative lookahead del matcher; (b) usar `node -e "fetch('http://localhost:3000/api/ping').then(r=>process.exit(r.ok?0:1))"` como test (node sí está en la imagen).

#### El contenedor corre como root, contradiciendo la documentación
- **Fichero:** `Dockerfile` (sin directiva `USER`), `.env.example:19`
- **Severidad:** 🟡 Media
- **Descripción:** `.env.example` afirma "El proceso del contenedor corre como UID 1000", pero el Dockerfile no tiene `USER node` — corre como root. Riesgo de seguridad (proceso con root escribiendo en volúmenes del NAS) y posible sorpresa de permisos al migrar.
- **Fix propuesto:** Añadir `USER node` antes del CMD (la imagen node ya trae el usuario `node` = UID 1000) y verificar permisos de `/data`.

#### Volúmenes correctos pero montaje `:ro` de todo el NAS
- **Fichero:** `docker-compose.yml:22-24`
- **Severidad:** 🟢 Baja
- **Descripción:** Lo esencial está bien: `/data` (BD + caché + backups) persistido, fotos en solo lectura. Montar `/volume1` completo `:ro` es más superficie de la necesaria — el contenedor puede leer todos los homes del NAS, no solo las fotos.
- **Fix propuesto:** Montar solo los directorios de catálogos necesarios (`/volume1/homes/javi/Photos:ro` + los de otros catálogos).
