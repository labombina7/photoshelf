# Tech Debt Audit — photoshelf
**Fecha:** 2026-05-23
**Estado:** pendiente

---

## Resumen ejecutivo

photoshelf es un proyecto joven con buena arquitectura base (Next.js App Router, Server Components,
SQLite con better-sqlite3, sharp para thumbnails). La deuda técnica está concentrada en cuatro áreas:
**código duplicado** (lógica de queries y helpers repetidos en múltiples archivos), **fragilidad de errores**
(rutas API sin try/catch), **ausencia de tests** (ningún módulo crítico tiene cobertura automatizada),
y **seguridad** (ya resuelta en PR #34). La deuda no bloquea el desarrollo actual pero se multiplicará
con cada nueva feature si no se aborda antes de EPIC-001 y EPIC-002.

---

## Hallazgos por categoría

### 🔁 Código duplicado

#### `upsertPhotoTags` — lógica DELETE+INSERT repetida ≥5 veces
- **Dónde:** `src/app/api/tags/[photoId]/route.ts`, `src/app/api/scan/route.ts`, otros
- **Situación actual:** El patrón de borrar tags existentes e insertar los nuevos se reimplementa en cada lugar donde se escriben tags.
- **Impacto:** Alto — cada bugfix requiere aplicarse en N sitios
- **Propuesta:** US-016 — extraer `upsertPhotoTags(db, photoId, tags)` a `src/lib/db-helpers.ts`

#### `buildPhotoQuery` — construcción de WHERE manual ≥4 veces
- **Dónde:** `src/app/api/photos/route.ts`, `src/app/api/timeline/route.ts`, `src/app/api/ai/search/route.ts`, otros
- **Situación actual:** Los filtros de fotos (year, event, tags, search) se construyen con string concatenation en cada endpoint.
- **Impacto:** Alto — bugs en filtros afectan a múltiples endpoints; EPIC-001 necesitará añadir `catalog_id` en todos
- **Propuesta:** US-016 — extraer `buildPhotoQuery(filters)` a un helper centralizado

#### `PHOTOS_PATH` definido en ≥7 archivos
- **Dónde:** Múltiples route handlers y lib files
- **Situación actual:** `process.env.PHOTOS_PATH ?? '/photos'` se copia en cada archivo que necesita la ruta.
- **Impacto:** Medio — ya parcialmente resuelto con `src/lib/config.ts` (US-015), pero algunos archivos no lo usan aún
- **Propuesta:** US-016 — migrar todos los usos a importar desde `src/lib/config.ts`

---

### 🚨 Fragilidad de errores

#### Rutas API sin try/catch
- **Dónde:** Varios archivos en `src/app/api/`
- **Situación actual:** Errores inesperados en route handlers (DB error, archivo no encontrado, Ollama timeout) pueden causar respuestas 500 sin información útil o que Next.js devuelva una página de error genérica.
- **Impacto:** Medio — dificulta el debugging en producción
- **Propuesta:** US-017 — pasada sistemática de try/catch + logging consistente

#### Tipado `any` en queries SQLite y respuestas Ollama
- **Dónde:** Queries `db.prepare(...).get()` casteadas como `any`, `data.response` de Ollama sin tipo
- **Situación actual:** El compilador TypeScript no puede detectar errores de acceso a propiedades inexistentes.
- **Impacto:** Medio — oculta bugs en refactoring
- **Propuesta:** US-017 — tipos explícitos para todas las rows de SQLite y respuestas de Ollama

---

### ⚡ Rendimiento

#### Queries N+1 en tags
- **Dónde:** Endpoints de timeline y library
- **Situación actual:** Se hace una query por foto para obtener sus tags, resultando en N+1 queries por página cargada.
- **Impacto:** Medio — en bibliotecas grandes (1000+ fotos) genera latencia notable
- **Propuesta:** US-018 — usar `GROUP_CONCAT` para incluir tags en la query principal

#### Polling de estado de scan cada 2 segundos
- **Dónde:** `src/components/ScanProgress.tsx`
- **Situación actual:** Mientras hay un scan activo, el cliente hace 30 peticiones por minuto al servidor.
- **Impacto:** Medio — carga innecesaria en el NAS 24/7
- **Propuesta:** US-018 — aumentar a 5s o implementar SSE

#### Sin índices en campos de filtrado frecuente
- **Dónde:** `src/lib/db.ts`
- **Situación actual:** `latitude`, `longitude` y posiblemente `year`, `event` no tienen índices, forzando full table scan.
- **Impacto:** Medio — se nota en bibliotecas grandes
- **Propuesta:** US-018 — añadir índices `IF NOT EXISTS` en el script de inicialización

---

### 🧪 Ausencia de tests

#### Cero tests en módulos críticos
- **Dónde:** `src/lib/scanner.ts`, `src/lib/thumbnail.ts`, `src/lib/ollama.ts`
- **Situación actual:** Los módulos más importantes del backend no tienen ningún test automatizado.
- **Impacto:** Alto — cada refactoring es un riesgo, los bugs sólo se detectan en producción
- **Propuesta:** US-019 — configurar Vitest + tests mínimos en módulos críticos

#### Sin tests de integración para rutas API
- **Dónde:** `src/app/api/`
- **Situación actual:** Los contratos de API (parámetros, respuestas, códigos HTTP) no están verificados automáticamente.
- **Impacto:** Medio — un cambio en la firma de una ruta puede romper la UI sin detectarse
- **Propuesta:** US-019 — tests de integración ligeros para las rutas más críticas

---

### 🏗️ Arquitectura

#### SQL inline en Server Components
- **Dónde:** `src/app/stats/page.tsx`, `src/app/library/page.tsx`
- **Situación actual:** Los Server Components hacen queries SQL directamente, mezclando la capa de datos con la de presentación.
- **Impacto:** Medio — dificulta la reutilización cuando llegue EPIC-002 (API iOS)
- **Propuesta:** US-022 — capa de repositorio en `src/lib/queries/`

#### Dependencia exclusiva de Ollama
- **Dónde:** `src/lib/ollama.ts`
- **Situación actual:** Si Ollama no está disponible, las funciones de IA fallan sin alternativa.
- **Impacto:** Bajo ahora, alto si se usa en contextos sin Ollama
- **Propuesta:** US-021 — sistema de adaptadores intercambiables (Ollama, OpenAI, Anthropic)

## Orden de ataque recomendado

```
1. US-016 (centralizar código duplicado) — prerequisito para todo lo demás
2. US-018 (rendimiento) — impacto inmediato en el NAS
3. US-017 (hardening técnico) — mejora resiliencia
4. US-022 (repositorio) — prerequisito de EPIC-002
5. US-019 (tests) — en paralelo con otras historias cuando hay capacidad
6. US-021 (multi-proveedor LLM) — cuando Ollama sea limitante
```
