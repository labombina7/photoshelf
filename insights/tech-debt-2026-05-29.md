## Hallazgos

### 🔒 Seguridad

#### Ausencia de rate limiting en endpoints de IA / operaciones pesadas
- **Fichero:** `src/app/api/ai/classify/batch/route.ts`, `src/app/api/ai/search/route.ts`, `src/app/api/ai/review/[photoId]/route.ts`
- **Severidad:** 🔴 Alta
- **Descripción:** Los endpoints de clasificación masiva y búsqueda profunda con Ollama no tienen protección contra abuso. Un usuario autenticado puede lanzar peticiones en bucle que bloqueen Ollama completamente, aumenten el consumo de CPU/RAM o provoquen denegación de servicio del contenedor. Solo el endpoint de login tiene rate limiting.
- **Fix propuesto:** Añadir un middleware o un guard HOF que limite a N peticiones simultáneas por endpoint (p. ej., un mutex en `classifyState.running` ya existe para classify, pero no para review ni para ai/search). Para ai/search implementar un throttle de 1 petición concurrente por sesión.

#### Catálogo con path arbitrario aceptado sin validación de accesibilidad real
- **Fichero:** `src/app/api/catalogs/route.ts:28–34`, `src/lib/queries/catalogs.ts:47–50`
- **Severidad:** 🔴 Alta
- **Descripción:** Al crear un catálogo, la validación de solapamiento de rutas se hace por comparación de strings, pero no se verifica que la ruta sea accesible. Un administrador puede crear un catálogo apuntando a `/etc`, `/proc` (en Linux) u otras rutas sensibles del sistema. El scanner leerá esas rutas y almacenará metadatos en la base de datos.
- **Fix propuesto:** En `createCatalog`, verificar que el path existe y es un directorio accesible antes de persistirlo, igual que hace `/api/scan/route.ts` (líneas 33–41). Además, validar que el path comienza con un prefijo permitido (por ejemplo, una variable de entorno `ALLOWED_ROOT`).

#### Middleware de autenticación solo cubre `/api/v1/*`
- **Fichero:** `src/middleware.ts:18–20`
- **Severidad:** 🔴 Alta
- **Descripción:** El middleware de Next.js aplica el check de sesión únicamente sobre `/api/v1/`. Todos los demás endpoints (`/api/photos/*`, `/api/ai/*`, `/api/scan`, etc.) dependen exclusivamente del check manual `getSession()` en cada route handler. Si un handler futuro olvidara el check, no habría ninguna barrera de contención. Además, el middleware solo comprueba la presencia de la cookie (no su validez criptográfica), por lo que un atacante podría fabricar una cookie con valor arbitrario y pasar el check del middleware.
- **Fix propuesto:** Ampliar el matcher del middleware a `/api/:path*` y promover la validación completa de iron-session al middleware para obtener una capa de defensa en profundidad. Los handlers individuales seguirían siendo la fuente de verdad, pero el middleware protegería rutas que se olviden de añadir el check.

#### Tamaño del body de la petición no limitado
- **Fichero:** `src/app/api/auth/login/route.ts:48`, `src/app/api/catalogs/route.ts:17`, `src/app/api/projects/generate/route.ts:54`
- **Severidad:** 🟡 Media
- **Descripción:** `req.json()` no está protegido por un límite explícito de tamaño. Next.js tiene un límite de 4MB por defecto en Route Handlers, pero puede ser configurado globalmente. Un cliente puede enviar un cuerpo de 4MB al endpoint de login o al de generación de proyectos, forzando un parse costoso y consumo de memoria.
- **Fix propuesto:** Leer el header `Content-Length` y rechazar con 413 si supera un umbral (p. ej., 10KB para login y 100KB para generate). Alternativamente, configurar `export const maxRequestBodySize = '10kb'` por route.

#### Aislamiento de la sesión compartida entre instancias / procesos
- **Fichero:** `src/lib/scanState.ts`, `src/lib/classifyState.ts`, `src/lib/watcherState.ts`
- **Severidad:** 🟡 Media
- **Descripción:** El estado de scan/classify/watcher se almacena en `globalThis` (memoria del proceso Node.js). En un entorno con múltiples workers o replicas (p. ej., PM2 cluster mode o Kubernetes con >1 pod), cada instancia tendría su propio estado, causando inconsistencias y condiciones de carrera. El folderwatcher también registra listeners de `fs.watch` por proceso.
- **Fix propuesto:** Para proyectos personales con un solo proceso esto es aceptable, pero debería documentarse explícitamente. Si se escala, usar una tabla SQLite para persistir el estado de operaciones en curso, que es nativo al stack ya existente.

### ⚠️ Gestión de errores

#### `ai/search` en modo deep silencia errores individuales sin logging
- **Fichero:** `src/app/api/ai/search/route.ts:78`
- **Severidad:** 🟡 Media
- **Descripción:** El bloque `catch { /* skip on error for individual photo */ }` no registra ningún log. En una sesión de búsqueda deep con 50 fotos, si Ollama falla en las 50, el usuario recibe un resultado vacío sin ningún indicio en los logs de qué sucedió.
- **Fix propuesto:** `catch (err) { console.error('[ai/search] photo error:', candidate.id, err instanceof Error ? err.message : err); }` — minimal logging que preserve la continuidad del bucle.

#### Catch vacío en `thumbnail.ts` al servir desde caché
- **Fichero:** `src/lib/thumbnail.ts:26`
- **Severidad:** 🟢 Baja
- **Descripción:** El bloque `try { buffer = await fs.readFile(cachePath); return ...; } catch {}` silencia cualquier error que no sea ENOENT (p. ej., EACCES, disco lleno) y continúa generando el thumbnail, enmascarando problemas de permisos en el directorio de caché.
- **Fix propuesto:** Distinguir entre `ENOENT` (cache miss normal) y otros errores que deben loguearse: `catch (err) { if ((err as NodeJS.ErrnoException).code !== 'ENOENT') console.error('[thumbnail] cache read error:', err); }`.

#### Catch vacío en `folderWatcher.buildDirSnapshot`
- **Fichero:** `src/lib/folderWatcher.ts:36`
- **Severidad:** 🟢 Baja
- **Descripción:** El bloque `try { ... } catch {}` externo en `buildDirSnapshot` silencia errores inesperados distintos de los `readdir/stat` que ya tienen `.catch()` individuales.
- **Fix propuesto:** `catch (err) { console.error('[watcher] buildDirSnapshot error:', err); }`.

#### `classify/year/route.ts` no usa `after()` — bloquea la respuesta durante hasta 300 s
- **Fichero:** `src/app/api/ai/classify/year/route.ts:59–81`
- **Severidad:** 🟡 Media
- **Descripción:** El route de classify/year lanza la clasificación en un IIFE `(async () => {})()` pero retorna la respuesta 202 inmediatamente. Sin embargo, el proceso de clasificación sigue vivo en un "fire and forget" no anclado al runtime de Next.js. A diferencia de `/api/scan/route.ts` que usa `after()` (que garantiza que el proceso no se corta), este handler usa un patrón no soportado que puede causar que la clasificación se aborte cuando el runtime de Next.js libera el contexto.
- **Fix propuesto:** Reemplazar el IIFE con `after(async () => { ... })` de next/server, igual que en `scan/route.ts`.

#### Error no capturado en `DetailPanel.removeTag` y `toggleTheme`
- **Fichero:** `src/components/DetailPanel.tsx:47–54, 104–115`
- **Severidad:** 🟢 Baja
- **Descripción:** Las funciones `removeTag` y `toggleTheme` llaman a `fetch` sin `try/catch` ni manejo de errores. Si la red falla o el servidor devuelve un error, el estado local del componente se actualizará de todas formas (optimistic update) pero sin notificar al usuario del fallo.
- **Fix propuesto:** Envolver las llamadas en try/catch y revertir el estado optimista en caso de error, o al menos mostrar un toast de error.

- **Fix propuesto:** Mover `getSidebarProjects` a `src/lib/queries/sidebar.ts` (o crear `queries/projects.ts`). Actualizar los dos importadores en `projects/page.tsx` y `projects/[id]/page.tsx`.

### 🚀 Rendimiento

#### Caché de thumbnails sin límite de tamaño en disco
- **Fichero:** `src/lib/thumbnail.ts:21`
- **Severidad:** 🟡 Media
- **Descripción:** Los thumbnails WebP generados se almacenan en `data/.cache/` sin ningún mecanismo de evicción. En una biblioteca grande (10.000+ fotos) con múltiples tamaños y fits (400-cover, 1920-inside, etc.), el directorio de caché puede crecer indefinidamente.
- **Fix propuesto:** Implementar un mecanismo LRU simple basado en `fs.stat` + `mtime`, o una tarea periódica (p. ej., integrada en el start del servidor) que elimine entradas con antigüedad > 30 días. Alternativamente, documentar un cron de limpieza en el `docker-compose` o `README`.

#### Consulta `COUNT(*)` en `listPhotos` no considera los filtros activos
- **Fichero:** `src/lib/queries/photos.ts:63`
- **Severidad:** 🟡 Media
- **Descripción:** `total` siempre devuelve el count total del catálogo (`SELECT COUNT(*) FROM photos WHERE catalog_id = ?`), independientemente de los filtros aplicados (año, evento, tag, favorito, etc.). Esto hace que la paginación sea incorrecta cuando hay filtros: `hasMore` y el total mostrado al usuario no reflejan la realidad.
- **Fix propuesto:** Calcular el `total` aplicando los mismos joins y conditions del filtro activo, o añadir una query separada de count con `buildPhotoFilter`.

#### `loadHints` en `execute.ts` carga todos los tags en cada búsqueda
- **Fichero:** `src/lib/search/execute.ts:55–69`
- **Severidad:** 🟡 Media
- **Descripción:** `loadHints` hace `SELECT name FROM tags ORDER BY name ASC` (todos los tags, sin límite) en cada llamada a `executeSearch`. En una biblioteca con miles de tags esto puede ser lento y produce arrays grandes en memoria solo para hacer matching de nombres.
- **Fix propuesto:** Cachear `hints` en memoria con una invalidación por tiempo (p. ej., 60 s) o por evento de escritura (añadir/eliminar tag). Alternativamente, limitar el número de tags cargados (p. ej., los 500 más usados).

#### `getMapPhotos` sin año carga hasta 5001 fotos de golpe
- **Fichero:** `src/lib/queries/photos.ts:261–276`
- **Severidad:** 🟢 Baja
- **Descripción:** Cuando se carga el mapa sin filtro de año, la query obtiene hasta 5001 filas (`MAP_LIMIT + 1 = 5001`) de la BD y las serializa todas en JSON. Para una biblioteca grande esto puede producir responses de varios MB.
- **Fix propuesto:** Implementar clustering del lado servidor (agrupar puntos GPS cercanos) o reducir `MAP_LIMIT`. A corto plazo, añadir `Content-Encoding: gzip` (ya habilitado por Next.js por defecto) y asegurarse de que la respuesta se comprime.

#### Polling a 5 s sin backoff en `ScanProvider` y `ClassifyProvider`
- **Fichero:** `src/components/ScanProvider.tsx:93`, `src/components/ClassifyProvider.tsx:63`
- **Severidad:** 🟢 Baja
- **Descripción:** Ambos providers hacen polling cada 5 s / 2 s respectivamente, siempre con el mismo intervalo. No hay backoff exponencial cuando no hay una operación en curso, causando hasta 720 peticiones por hora al servidor, incluso cuando el usuario está inactivo en la página.
- **Fix propuesto:** Cuando `running === false`, ampliar el intervalo a 30 s o dejar de hacer polling hasta que el usuario inicie una operación. Usar `visibilitychange` para pausar el polling cuando la pestaña está en background.

### 🏗️ Arquitectura

#### `getDb()` importado directamente en route handlers y pages — viola la capa de repositorio
- **Fichero:** `src/app/api/search/hints/route.ts`, `src/app/api/search/suggestions/route.ts`, `src/app/api/projects/route.ts`, `src/app/api/projects/generate/route.ts`, `src/app/api/projects/[id]/route.ts`, `src/app/api/ai/search/route.ts` y 6 más
- **Severidad:** 🟡 Media
- **Descripción:** La guía de arquitectura en `CLAUDE.md` establece "Nunca importar `getDb()` directamente en pages o route handlers nuevos — usar funciones del repositorio en `src/lib/queries/`". Sin embargo, al menos 12 ficheros de routes/pages importan `getDb` directamente y ejecutan SQL inline. Esto dispersa la lógica de acceso a datos, dificulta los tests y hace la refactorización del esquema más costosa.
- **Fix propuesto:** Crear funciones de repositorio para: `getProjectList`, `getProjectById`, `getSearchHints`, `getSearchSuggestions`, `getAiSearchCandidates`, etc. Mover el SQL inline a `queries/projects.ts` y `queries/search.ts`.

#### Endpoint `ai/search` marcado como `@legacy` sin fecha de resolución
- **Fichero:** `src/app/api/ai/search/route.ts:1–5`
- **Severidad:** 🟡 Media
- **Descripción:** El comentario `@legacy — Endpoint mantenido para el modo deep de US-032` indica deuda intencional sin fecha de resolución ni historia de usuario asignada. El endpoint tiene duplicación de lógica con `executeSearch` (catalog_id no se usa en el modo deep).
- **Fix propuesto:** Crear US en el backlog para integrar el modo deep en `/api/search?mode=deep`. Mientras tanto, añadir el catalog_id de la sesión al modo deep (bug: actualmente busca en todos los catálogos).

#### Estado de clasificación/scan acoplado a `globalThis` — no hay invalidación ante reinicios
- **Fichero:** `src/lib/scanState.ts`, `src/lib/classifyState.ts`, `src/lib/watcherState.ts`
- **Severidad:** 🟢 Baja
- **Descripción:** Si el proceso Node.js se reinicia durante una operación (p. ej., hot reload en dev, crash en prod), el estado queda atascado en `running: true` hasta el siguiente reinicio limpio. No hay reconciliación al arrancar.
- **Fix propuesto:** En la inicialización de `globalThis`, forzar `running: false` y `completedAt: Date.now()` para que el estado siempre arranque limpio. Esto ya está parcialmente implementado (los objetos iniciales tienen `running: false`), pero no se resetea si el proceso muere con `running: true` persistido.

#### v1 original duplicado entre `/api/photos/*` y `/api/v1/photos/*`
- **Fichero:** `src/app/api/photos/`, `src/app/api/v1/photos/`
- **Severidad:** 🟡 Media
- **Descripción:** Existe duplicación de endpoints para fotos: `/api/photos/[id]` y `/api/v1/photos/[id]`, `/api/photos/[id]/thumbnail` y `/api/v1/photos/[id]/thumbnail`, etc. Los de v1 son la versión canónica (con `withAuth` HOF y `apiSuccess`/`apiError`), pero los originales siguen activos y siendo usados por el frontend. Además, `/api/v1/photos/[id]/original/route.ts` define `MIME_TYPES` localmente duplicando `config.ts` (viola la regla del CLAUDE.md).
- **Fix propuesto:** Migrar progresivamente el frontend a `/api/v1/*` y deprecar la ruta antigua, o unificar en un solo conjunto. Corregir la duplicación de `MIME_TYPES` en `/api/v1/photos/[id]/original/route.ts` importando desde `@/lib/config`.

### 🔧 Mantenibilidad

#### `ollama.ts` supera 300 líneas con 5 responsabilidades distintas
- **Fichero:** `src/lib/ollama.ts` (341 líneas)
- **Severidad:** 🟡 Media
- **Descripción:** El módulo `ollama.ts` contiene: extracción de JSON, escape de XML, lectura de imagen, comunicación HTTP con Ollama (text y vision), clasificación de fotos, parsing de búsquedas, generación de proyectos y review de fotos. Es el módulo más complejo y difícil de testear del proyecto.
- **Fix propuesto:** Dividir en submódulos: `src/lib/ollama/client.ts` (HTTP), `src/lib/ollama/image.ts` (lectura + resize), `src/lib/ollama/classify.ts`, `src/lib/ollama/search.ts`, `src/lib/ollama/review.ts`, `src/lib/ollama/projects.ts`.

#### `PhotoDetailClient.tsx` supera 340 líneas con lógica compleja de touch/HUD
- **Fichero:** `src/app/library/[photoId]/PhotoDetailClient.tsx` (340 líneas)
- **Severidad:** 🟡 Media
- **Descripción:** El componente gestiona simultáneamente: sidebar desktop/mobile, HUD mobile fullscreen, swipe horizontal, navegación por teclado, contador de fotos, lógica de touch para tablet vs. móvil. Sería difícil añadir nueva funcionalidad (p. ej., zoom de imagen) sin aumentar la complejidad.
- **Fix propuesto:** Extraer la lógica de touch/swipe a un hook `usePhotoSwipe(prevId, nextId, navSearch)`. Extraer el HUD mobile a un componente separado `PhotoViewerMobile`.

#### Magic numbers en configuración de thumbnails y timeouts
- **Fichero:** `src/lib/ollama.ts:63,91,197`, `src/lib/thumbnail.ts:6`, `src/lib/folderWatcher.ts:10,11`, `src/app/api/auth/login/route.ts:6,7`
- **Severidad:** 🟢 Baja
- **Descripción:** Constantes como `30_000`, `120_000`, `180_000` (timeouts de Ollama), `5_000`, `30_000` (debounce/poll del watcher), `512` (tamaño de resize para Ollama), `10`, `15 * 60 * 1000` (rate limiter) están dispersas sin nombres descriptivos.
- **Fix propuesto:** Centralizar en `src/lib/config.ts` o en un archivo `src/lib/constants.ts`: `export const OLLAMA_TIMEOUT_TEXT_MS = 30_000`, `OLLAMA_TIMEOUT_VISION_MS = 120_000`, `OLLAMA_IMAGE_SIZE = 512`, `WATCHER_DEBOUNCE_MS = 5_000`, etc.

#### `console.log` para logging de operaciones en producción
- **Fichero:** `src/lib/folderWatcher.ts:55,72,123,132,146,153,173`, `src/lib/scanner.ts:142`, `src/lib/db.ts:155,204`
- **Severidad:** 🟢 Baja
- **Descripción:** Se usa `console.log` mezclado con `console.error` para logging de nivel informativo. Sin una librería de logging estructurado, no hay distinción de niveles y no se puede filtrar en producción.
- **Fix propuesto:** A corto plazo, unificar en `console.info` para mensajes informativos y `console.error` para errores. A largo plazo, considerar una librería mínima como `pino` para logging estructurado con niveles y formato JSON en producción.

- **Fichero:** `src/app/api/**`, `src/components/**`, `src/app/library/**`

No existe ningún test de integración para route handlers, ni tests de componentes con `@testing-library/react` (que sí está instalado como devDependency). Los routes críticos como `scan`, `classify/batch`, `catalogs` o `auth/login` no tienen cobertura.

Esto significa que si la lógica de parsing cambia en `ollama.ts`, los tests no detectarán la regresión.

#### `auth/login/route.ts` tiene un directorio `__tests__` vacío o sin tests reales

- **Fix propuesto:** Asegurar tests para: (1) login correcto, (2) password incorrecto, (3) rate limit (429), (4) body sin campo `password`, (5) `APP_PASSWORD` no configurado.

### 📦 Dependencias

#### `node-addon-api` como dependencia de producción innecesaria
- **Fichero:** `package.json:25`
- **Severidad:** 🟡 Media
- **Descripción:** `node-addon-api` es una dependencia de compilación nativa para módulos N-API, utilizada internamente por `better-sqlite3` y `sharp`. No debería estar en `dependencies` de la aplicación, sino gestionarse a través de las dependencias transitivas. Incluirla en `dependencies` la envía al bundle de producción/standalone sin necesidad.
- **Fix propuesto:** Mover `node-addon-api` a `devDependencies` o eliminarla completamente (ya viene como dependencia transitiva de `better-sqlite3`). Verificar que el build de Docker sigue funcionando tras el cambio.

#### `leaflet.markercluster` sin versión compatible con Leaflet 2.x (si se actualiza)
- **Fichero:** `package.json:22`
- **Severidad:** 🟢 Baja
- **Descripción:** `leaflet@1.9.4` es la versión estable actual. `leaflet.markercluster@1.5.3` soporta Leaflet 1.x. Si se actualiza Leaflet a una versión 2.x futura, habrá breaking changes en la integración. No es un problema hoy, pero es un riesgo a documentar.
- **Fix propuesto:** Documentar la compatibilidad en un comentario en `package.json` o en el CLAUDE.md. Monitorizar el release de Leaflet 2.x.

#### Versiones de `@types/node` y `typescript` sin pinning exacto
- **Fichero:** `package.json:30,35`
- **Severidad:** 🟢 Baja
- **Descripción:** `@types/node: "^20"` y `typescript: "^5"` usan rangos amplios. Un `npm install` en un entorno diferente puede traer una versión de TypeScript con breaking changes en el compilador, o tipos de Node que rompan el build.
- **Fix propuesto:** Pinear las versiones exactas: `"@types/node": "20.x.x"`, `"typescript": "5.x.x"`. Usar `package-lock.json` (ya existe) y asegurarse de que el CI/CD siempre usa `npm ci`.
